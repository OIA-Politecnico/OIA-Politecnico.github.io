## A - Seats

El problema nos pide obtener la mínima cantidad **total** de alumnos sentados tal que sea imposible sentar algún otro alumno *sin que esté al lado de otro ya sentado*. Como no podemos levantar alumnos ya sentados, nuestro objetivo es minimizar la cantidad de nuevos alumnos que sentamos en los espacios vacíos, tal que no se rompa la condición mencionada.

Como el problema nos pide calcular una cantidad, no es necesario devolver *cómo* se sentarían los alumnos en una configuración óptima. Por lo tanto, es preferible obtener una cuentita matemática general que nos permita obtener la respuesta "para cualquier caso" o cuentitas que cubran "todos los casos".

El enunciado nos asegura que no hay dos alumnos sentados uno al lado de otro, por lo que la string que nos pueden dar es de la siguiente forma:

```
[1] ...000... 1 ...000... 1 ...000... [...] [1]
```

Es decir, conjuntos de `0`s consecutivos separados por un solo `1` entre sí, y posiblemente en el inicio y final de la string.

Dado que solo nos interesan los espacios vacíos, podemos pensar en cada secuencia de `0`s consecutivos por separado. En efecto, cada una de estas secuencias es un problema aislado. Si sentamos la menor cantidad de alumnos en una de estas secuencias o *bloques* de `0`s, esta cantidad no influye la menor cantidad que podríamos sentar en otro bloque. Esto se debe a que dos bloques, por la forma en que los separamos, están separados al menos por un `1` entre ellos. De esta forma poner alumno en un bloque no influye en el otro, ya que ninguno de los alumnos que sentemos en el primero se encontrará adyacente a un alumno sentado en el otro (y viceversa):

```
...00010000AB1CD00001000...
```

En el ejemplo anterior, incluso si se sienta un nuevo alumno en la posición `A`, éste no está ni cerca de la posición `D`, que sería la menor posición a la que se puede sentar un alumno en el segundo bloque. Notar que no se puede sentar a un alumno en `B` ni `C` porque estaría al lado de un alumno ya sentado de antes, representado con un `1`.

Trabajemos entonces con bloques de `0s`. Si bien son independientes entre sí, es importante observar que no son todos iguales. Por ejemplo, la respuesta para el siguiente bloque de cinco `0`s:

```
...1000001...
```

es distinto que para el bloque:

```
...100000 (la string termina acá)
```

En el primer caso, basta con poner un alumno en la posición del medio, ya que así el resto de posiciones vacías estarían adyacentes a un alumno ya sentado:

```
...100 *1* 001...
```

Mientras que en el segundo bloque visto, necesitaríamos sentar al menos dos alumnos. Esto pasa porque, incluso si sentamos a uno en el medio igual que antes, todavía podemos sentar a otro en el extremo derecho, porque **no tiene a un alumno sentado a la derecha**:

```
...100 *1* 0 *1* (la string termina acá)
```

otra solución podría ser:

```
...10 1* 00 *1* (la string termina acá)
```

En general, podemos distinguir tres casos:

1) Un bloque de `0`s con dos alumnos sentados a ambos lados. Por ejemplo:

```
...10001...
```

2) Un bloque de `0`s con solo un alumno sentado a la izquierda. Por ejemplo:

```
...1000 (fin de la string)
```

3) Un bloque de `0`s con un solo alumno sentado a la derecha. Por ejemplo:

```
(inicio de la string) 0001...
```

Como se pudo ver, los últimos dos casos solo se podrían presentar al inicio y al final de la string. Es decir, estos casos **no** se pueden presentar en el medio, debido a la definición de bloque que utilizamos (cada uno de las secuencias de `0`s están separadas por `1`s, o bien se encuentran al principio o final de la string).

> Notar que un caso de ejemplo posible podría ser este:
>
> ```
> (inicio string) 10001 (final string)
> ```
>
> En este caso, el único bloque no tiene el principio ni el final de la string, pero tampoco tiene bloques de `0`s antes ni después. Sin embargo, lo que nos importa es que tiene `1`s a ambos lados.

Por lo tanto, para ayudar a nuestro razonamiento, podríamos ver de **_convertir_** esos dos últimos casos en en el primero. Veamos, por ejemplo, el caso que la string empieza con un solo `0`:

```
01...
```

En ese caso, no se puede sentar un alumno ahí porque está al lado de otro ya sentado. Consideremos ahora si hay dos `0`s:

```
001...
```

En ese caso, podemos debemos sentar un alumno en la primera posición:

```
*1* 01...
```

Supongamos ahora que empieza con tres `0`s:

```
0001...
```

Para resolver este caso, podríamos sentar un alumno tanto en la primera como en la segunda posición, ya que en ambos casos el resto de espacios libres del bloque serían adyacentes a al menos un alumno ya sentado:

```
*1* 001...
```
```
0 *1* 01...
```

¿Y con 4 `0`s?

```
00001...
```

En este caso **no** podemos poner un alumno en la primera posición, ya que así quedaría un espacio libre para que otro alumno `A` pueda sentarse:

```
*1* 0A01...
```

En cambio, la solución que minimiza la cantidad de nuevos alumnos a sentar en ese bloque es sentar uno en la *segunda* posición:

```
0 *1* 001...
```

Luego, si la string empieza con **cuatro o más** `0`s, siempre conviene poner un alumno en la segunda posición, y luego podemos tratar el bloque después de ese alumno como *el caso 1)* que vimos antes:

```
0 *1* [000000] 1 ...
```

Entre corchetes [] se encuentra un bloque de `0`s con alumnos sentados a ambos lados, el caso 1).

El mismo razonamiento valdría para cuando la string *termina* con `0`s. Por ejemplo:

```
...10000 (fin string)
```

En conclusión, podemos primero transformar la string en una con todos bloques con `1`s a ambos lados bajo la siguiente idea:

> Si empieza o finaliza con `0`s, según la cantidad que sean:
> * **un `0`**: ignoro el bloque
> * **dos `0`s**: sumo 1 a la respuesta e ignoro el bloque
> * **tres `0`s o más**: sumo 1 a la respuesta, y considero el bloque sin los primeros (o últimos, según corresponda) dos dígitos

En código, primero contamos la cantidad de `0`s al principio y al final:

```cpp
int cntini = 0;
for (int i = 0; i < n; i++) {
    if (s[i] == '0') {
        cntini++;
    }
    else { // == '1'
        break;
    }
}

int cntfin = 0;
for (int i = n-1; i >= 0; i--) {
    if (s[i] == '0') {
        cntfin++;
    }
    else { // == '1'
        break;
    }
}
```

Y luego aplicamos las reglas anteriores. `ini` y `fin` serán los extremos de la string que consideraremos para la próxima parte:

```cpp
int rta = 0;
int ini = 0, fin = n-1;
if (cntini == 1) {
    ini = 1;
}
else if (cntini == 2) {
    ini = 2;
    rta++;
}
else { // >= 3
    ini = 2;
    rta++;
}

if (cntfin == 1) {
    fin = n-2;
}
else if (cntini == 2) {
    fin = n-3;
    rta++;
}
else { // >= 3
    fin = n-3;
    rta++;
}
```

Observando que siempre sumamos o restamos a `ini` / `fin` la misma cantidad de `0`s contada en el extremo (hasta 2), se reduce a:

```cpp
int rta = 0;
int ini = min(2,cntini), fin = n-1-min(2,cntfin);

if (cntini >= 2) {
    rta++;
}
if (cntfin >= 2) {
    rta++;
}
```

Luego, iteramos desde `ini` hasta `fin` descubriendo los bloques (todos con `1`s a ambos lados):

```cpp
int cnt = 0;
for (int i = ini; i <= fin; i++) {
    if (s[i] == '0') {
        cnt++;
    }
    else {
        rta += // ???
        cnt = 0;
    }
}
```

Nos queda determinar cuál es la mínima cantidad de alumnos que se pueden sentar en un bloque de `0`s con `1` sentados a ambos lados, tal que no se puedan sentar más alumnos.

Observemos, en primer lugar, que en ambos extremos no se puede sentar a nadie, porque ambas posiciones son adyacentes a otro alumno (marcadas con `X`):

```
...1X000X1...
```

Siempre que podamos, sentaremos en los `0`s del medio a alumnos lo más lejos entre sí, a fin de minimizar la cantidad de alumnos nuevos a sentar:

* Con uno y dos `0`s no hace falta sentar a nadie:

```
...101...
```
```
...1001...
```

* Con tres `0`s necesariamente sentamos uno en el medio:

```
...10 *1* 01...
```

* Con cuatro, basta con sentar a uno en alguno de las dos posiciones del medio:

```
...10 *1* 001...
```
```
...100 *1* 01...
```

* Con cinco, basta con poner uno en el medio:

```
...100 *1* 001...
```

* Con seis ya necesitamos dos personas. En efecto, si ponemos un solo alumno, siempre las posiciones marcadas con `D` están disponibles:

```
...10 *1* 0DD01...
```
```
...100 *1* 0D01...
```
```
...10D0 *1* 001...
```
```
...10DD0 *1* 01...
```

Entonces una solución sería:

```
...100 *1* 0 *1* 01...
```

En general, el problema está en que no queden **tres `0`s** consecutivos, porque eso permite que haya una posición disponible. Si hay dos consecutivos, ambas posiciones tán inhabilitadas por los `1`s a sus lados.

Así, si tenemos *n* `0`s, la mínima cantidad de alumnos a sentar por bloque es:

* `n=1` -> 0
* `n=2` -> 0
* `n=3` -> 1
* `n=4` -> 1
* `n=5` -> 1
* `n=6` -> 2
* ...

Es decir, **la respuesta es `n/3`**.

Reemplazando en el código de antes:

```cpp
rta += cnt / 3;
```

Finalmente, hay que tener cuidado que al final, según la lógica de antes siempre `fin` está sobre un dígito `0`, por lo que la rama del else no se ejecuta. Es decir, al terminar el `for`, hay que sumar a la respuesta el último bloque.

***¿Y si no hay ni un `1`?*** Ese es otro caso a tener en cuenta. Con la lógica de antes, debería funcionar bien si `n >= 5`, ya que nuestro código consideraría que tiene un bloque de `0`s al inicio al final de más de tres `0`, considerando una solución como esta: `00000` -> `01 000` -> `01 0 10`. Notar que esta solución es óptima ya que no podemos poner un `1` en el medio, porque en ese caso ambos extremos estarían disponibles para que un alumno se siente: `D0 *1* 0D`.

Por lo tanto, basta con considerar aparte los casos de uno a cuatro `0`s:

* `0` -> `1` (1 alumno)
* `00` -> `01` o `10` (1 alumno)
* `000` -> `010` (1 alumno)
* `0000` -> `1001` (2 alumnos)

> Notar que de hecho nuestra solución de antes también funciona para `n >= 4`. Si bien nuestra lógica consideraría algo como `0000` -> `0110` que sería inválido porque hay dos alumnos sentados, lo que nos interesa es nomás la cantidad de alumnos nuevos a sentar, que son también solo dos.

Así, debemos agregar antes en el código:

```cpp
if (cntini == n && n <= 3) {
    cout << "1\n";
    continue; // siguiente test
}
```

También debemos pensar si nuestra lógica funciona en los casos chicos cuando sí hay algún `1`. Es decir, cuando `n <= 3` pero hay algún `1`. Estos casos son pocos:

* `101`
* `010`
* `100`
* `001`
* `10`
* `01`
* `1`

Se deja como ejercicio pensar que estos casos siempre funcionan con la solución planteada.

Combinando todo, obtenemos Accepted!

> **IMPORTANTE**: Recordar que la solución nos pide la cantidad *total* de alumnos sentados, por lo que debemos sumar los `1` que ya están de antes en la string.

```cpp
#include<bits/stdc++.h>
 
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int t; cin >> t;
    for (int tt = 0; tt < t; tt++) {
        int n; cin >> n;
        string s; cin >> s;

        int cntunos = 0;
        for (int i = 0; i < n; i++) {
            if (s[i] == '1') {
                cntunos++;
            }
        }

        int cntini = 0;
        for (int i = 0; i < n; i++) {
            if (s[i] == '0') {
                cntini++;
            }
            else { // == '1'
                break;
            }
        }

        int cntfin = 0;
        for (int i = n-1; i >= 0; i--) {
            if (s[i] == '0') {
                cntfin++;
            }
            else { // == '1'
                break;
            }
        }

        if (cntini == n && n <= 3) {
            cout << "1\n";
            continue; // siguiente test
        }

        int rta = 0;
        int ini = min(2,cntini), fin = n-1-min(2,cntfin);

        if (cntini >= 2) {
            rta++;
        }
        if (cntfin >= 2) {
            rta++;
        }

        int cnt = 0;
        for (int i = ini; i <= fin; i++) {
            if (s[i] == '0') {
                cnt++;
            }
            else {
                rta += cnt/3;
                cnt = 0;
            }
        }
        rta += cnt/3;

        cout << rta + cntunos << '\n';
    }
    return 0;
}
```

## B - Ultimate Figure Skating (difícil, nivel 3)

> Dado que el contest estaba armado para todos, incluímos algunos problemas que deberían resultar interesantes para participantes de nivel 3. Como este es un problema para participantes más avanzados, la descripción de su solución será mucho más breve a la del problema anterior.

> Para resolver este problema se requiere conocer sobre SCCs (componentes fuertemente conexas). Si no conocés del tema, podés ver más info [acá](https://cp-algorithms.com/graph/strongly-connected-components.html).

En primer lugar, se puede modelar el problema como un grafo, donde la arista `x -> y` indica que el truco `y` debe realizarse después de `x`. Por lo tanto, como toda la cadena de trucos que uno haga debe seguir estas reglas, el objetivo es encontrar un camino en el grafo cuya suma de los puntos correspondientes a los nodos (trucos), sin repetir, sea máxima.

Si el grafo fuera acíclico, o sea, un DAG (*directed acyclic graph*), podríamos calcular el resultado con una DP, utilizando el *orden topológico* de los nodos. Sin embargo, no hay ninguna restricción de este tipo en el enunciado.

Sin embargo, observar que si uno de los nodos dentro de un ciclo forma parte del camino de mayor valor, deben pertenecer entonces **todos** los nodos del ciclo. En efecto, como el valor de cada truco se considera *solo una vez* y los valores de cada truco son *positivos*, conviene recorrer todo el ciclo **siempre**. Hacer esto suma mayor puntaje, y no impide de volver al nodo original, por lo que no se descartan "posibles caminos" que puedan cambiar el valor total máximo más adelante.

Por lo tanto, conviene considerar a todos los nodos de un ciclo juntos. De forma más general, consideraremos todos los nodos de una *componente fuertemente conexa* juntos. Al fin y al cabo, como todos los nodos de una SCC son alcanzables entre sí, podemos aplicar un razonamiento similar al anterior para determinar que si uno de ellos pertenece a la solución, entonces pertenecerán todos los de la SCC a la que pertenece.

Para considerar las nodos de cada SCC "juntos" podemos construir lo que se conoce como el *grafo de condensación* del grafo dado por la entrada. Este grafo se construye del original agrupando los vértices dentro de una misma SCC en un solo nodo, y conectando estos nodos mediante aristas según el grafo original. Así, podemos asignar a cada nodo de este nuevo grafo un valor correspondiente a la suma de los valores de todos los nodos que agrupa.

Este nuevo grafo será, entonces, un DAG. En efecto, si tuviera un ciclo, todos los nodos pertenecerían a una SCC (porque serían todos alcanzables entre sí), debiendo entonces estar agrupados en un solo nodo. Esto contradice por construcción que cada uno de los nodos son SCCs separadas (en el grafo de condensación).

Finalmente, podemos aplicar DP sobre el nuevo grafo, recorriendo los nodos en su orden topológico (*topological sort*). Para cada nodo se puede calcular el valor máximo total de los caminos que llegan a ese nodo. De esta forma, cuandp se elimina un nodo de la cola en el toposort, se debe de actualizar el valor de la DP en los extremos de las aristas que parten de ese nodo. El resultado será el máximo entre todos los valores de la DP para cada uno de los nodos, sumado al valor (total de los nodos del grafo original que abarca) de ese nodo.

> Para más detalles sobre el grafo de condensación, ver el artículo de cp-algorithms mencionado al principio.

> Este problema es idéntico a [este](https://cses.fi/problemset/task/1686) problema de CSES.

```cpp
#include<bits/stdc++.h>
 
#define forn(i,n) for(int i = 0; i < int(n); i++)
#define forsn(i,s,n) for(int i = int(s); i < int(n); i++)
#define dforn(i,n) for(int i = int(n)-1; i >= 0; i--)
#define dforsn(i,s,n) for(int i = int(n)-1; i >= int(s); i--)
#define fst first
#define snd second
#define pb push_back
#define all(c) (c).begin(),(c).end()
#define FAST_IO ios::sync_with_stdio(false);cin.tie(nullptr)
 
using namespace std;
typedef vector<int> vi;
typedef long long ll;
typedef pair<int,int> ii;
typedef long double ld;

int const MAXN = 1e5+5;
int const MAXM = 2e5+5;

struct edge {
    int v,c;
};

int p[MAXN];
int tin[MAXN], actT = 1;
bitset<MAXN> matched;
vi G[MAXN], G2[MAXN];
int comp[MAXN], comps = 0;
int indeg[MAXN];
ll sum_comp[MAXN], dp[MAXN];

vi pila;
int tarjan (int st) {
    tin[st] = actT++;
    pila.pb(st);

    int mini = tin[st];
    for (auto &i : G[st]) {
        if (!tin[i]) mini = min(mini, tarjan(i));
        if (!matched[i]) mini = min(mini, tin[i]);
    }

    if (mini >= tin[st]) {
        while (!pila.empty()) {
            auto e = pila.back(); pila.pop_back();
            matched[e] = true;
            comp[e] = comps;
            sum_comp[comps] += p[e];
            if (e == st) {comps++; break;}
        }
    }

    return mini;
}


int main() {
    FAST_IO;

    int n,m; cin >> n >> m;

    forn(i,n) cin >> p[i];

    forn(i,m) {
        int x,y; cin >> x >> y; --x, --y;
        G[x].pb(y);
    }

    forn(i,n) if (!tin[i]) tarjan(i);

    // condensed
    forn(i,n) for (auto &j : G[i]) if (comp[i] != comp[j]) G2[comp[i]].pb(comp[j]), indeg[comp[j]]++;

    ll maxi = 0;

    // toposort
    queue<int> Q;
    forn(i,comps) if (!indeg[i]) Q.push(i);
    while (!Q.empty()) {
        auto v = Q.front(); Q.pop();
        dp[v] += sum_comp[v];
        maxi = max(maxi,dp[v]);
        for (auto &i : G2[v]) {
            dp[i] = max(dp[i], dp[v]);
            if (!(--indeg[i])) {
                Q.push(i);
            }
        }
    }

    cout << maxi << '\n';

    return 0;
}
```

## C - Tab Closing

> Antes de resolver este problema, se recomienda utilizar el visualizador incluido en el enunciado probando (al menos) los casos de prueba de ejemplo, intentando entender cada uno de los valores de la respuesta.

Arrancamos la explicación de la solución con una observación:

> **Observación 1**: Siempre deberemos mover el mouse **al menos** una vez.

Si leemos detenidamente el enunciado, se nos indica que el cursor se encuentra inicialmente en el *extremo más izquierdo* de la pantalla. Es decir, si entendemos a la pantalla como una línea de longitud \\(a\\) como indica el enunciado, el cursor se encuentra en el extremo izquierdo de la línea.

Por lo tanto, como \\(n \geq 1\\) en el enunciado, siempre deberemos mover el cursor *al menos una vez* para cerrar alguna de las pestañas. Esto se debe a que la cruz (X) para cerrar cada una siempre se encuentra en el extremo *derecho* de cada pestaña. Por lo tanto, por más corta sea la longitud de cada una, como su longitud es siempre \\(\min(b,\frac{a}{m})\\) (el *mínimo* entre `b` y `a/m`), con \\(a,b \geq 1\\), nunca será cero. Por ende, la distancia hacia la primera de las cruces siempre será positiva, requiriendo mover el cursor hacia la derecha *al menos* una vez.

> **Observación 2**: Mientras la longitud de las ventanas se mantenga igual, no hará falta mover el cursor (excepto que cerremos la última).

Esto es fácil de ver, ya que si cerramos una pestaña que no sea la última, y el resto de pestañas no cambian de tamaño al hacerlo, quedará la cruz de la próxima ventana en la hilera directamente sobre bajo el cursor. Es decir, al cerrar una pestaña, todas las pestañas siguientes se desplazarán hacia la izquierda, de forma que la primera de ellas toma el lugar de la anteriormente cerrada.

> **Observación 3**: Si se cierra una pestaña y el resto **no** cambia de tamaño, entonces nunca más cambiarán de tamaño de ahora en adelante.

Esta observación se desprende directamente de la fórmula de la longitud de una pestaña. Como \\(a\\) y \\(b\\) son fijos, si se cierra una pestaña, la cantidad de pestañas restantes \\(m\\) siempre va a disminuir. En consecuencia, la división \\(\frac{a}{m}\\) cambiará. Sin embargo, si la longitud de las pestañas no cambió, quiere decir que \\(\text{len} \neq \frac{a}{m}\\), pues sabemos que \\(\frac{a}{m}\\) cambió. Necesariamente debe ser que \\(\text{len} = b\\). Eso implica que \\(\frac{a}{m} < b\\). Luego, como \\(m\\) seguirá disminuyendo a medida que cerremos más pestañas, \\(\frac{a}{m}\\) se volverá cada vez más grande, manteniéndose la condición \\(\frac{a}{m} < b\\). Por lo tanto, la condición \\(\text{len} = b\\) se seguirá cumpliendo hasta cerrar **todas** las pestañas.

Con esto en cuenta, sigue inmediatamente la siguiente observación.

> **Observación 4**: Apenas vale que \\(\frac{a}{m} \leq b\\), conviene mover el cursor para cerrar la primera de las pestañas.

En efecto, si apenas vale \\(\frac{a}{m} \leq b\\) comenzamos a cerrar las pestañas desde la primera de todas, por la *Observación 3* ya no deberemos mover el cursor hasta cerrar las pestañas que quedan, pues las restantes se irán deslizando para reeemplazar las cerradas hasta que no quede ninguna. Esto siempre conviene, incluso si cuando se cumplió la condición el cursor quedó listo para cerrar otra pestaña, ya que si no está posicionado sobre la primera de todas, después hará falta mover el cursor para cerrar las que quedan (eventualmente cerraremos todas hasta la última, pero quedarán pestañas a la izquierda por cerrar). Es decir, lo que nos ahorramos evitando mover el cursor en un momento, requerirá más tarde moverlo de todas formas.

Es importante observar que la condición usa \\(\leq\\) y no \\(<\\), pues apenas la división \\(\frac{a}{m}\\) sea igual a \\(b\\), el valor de \\(\text{len}\\) seguirá siendo \\(b\\), por lo explicado en la *Observación 3*.

> **Observación 5**: Si no vale \\(\frac{a}{m} \leq b\\), es decir, si \\(\frac{a}{m} > b\\), entonces conviene cerrar todas las pestañas desde el final (extremo derecho).

Cuando la barra está "llena de pestañas" y vale \\(\frac{a}{m} > b\\), resulta por definición de \\(\text{len}\\) que \\(\text{len} = \frac{a}{m}\\). Luego, como hay \\(m\\) pestañas restantes, las cruces de cada una de las pestañas se encuentran en posiciones \\(\frac{a}{m}, 2\frac{a}{m}, \dots m\frac{a}{m}\\). Esto es, la cruz de la última pestaña se encuentra en el extremo derecho de la pantalla, pues \\(m\frac{a}{m} = a\\). Esta condición valdrá siempre que \\(\text{len} = \frac{a}{m}\\). Por lo visto en la *Observación 3*, seguirá siendo así hasta que \\(m\\) sea lo suficientemente chico como para que se cumpla \\(\frac{a}{m} \leq b\\). En ese caso convendrá mover el cursor a la cruz de la primera pestaña.

Sin embargo, mientras no se cumpla, al cerrar una pestaña, el tamaño del resto de las pestañas se expandirá tal que la cruz de la *nueva* última pestaña quede también sobre el extremo derecho de la pantalla. Por ese motivo, convendrá tener el mouse en el extremo derecho hasta que se cumpla la condición de la *Observación 4*.

Finalmente, surge la última observación.

> **Observación 6**: Solo hará falta mover el mouse **a lo sumo** dos veces.

Si al comenzar a cerrar pestañas se cumple la condición de la *Observación 4*, entonces convendrá mover una sola vez el mouse hasta la cruz de la primera pestaña, permitiendo luego cerrar todas sin más movimientos.

Caso contrario, significa que \\(\frac{a}{m} > b\\), por la *Observación 5*. En consecuencia, convendrá mover el cursor hasta la cruz de la última pestaña, cerrando todas hasta que se cumpla la condición de la *Observación 4*, en cuyo caso convendrá mover el mouse hasta la cruz de la primer pestaña, cerrando las restantes. En total, se moverá el mouse solo dos veces.

> Notar que puede suceder que \\(\frac{a}{1} > b\\), por lo que el segundo movimiento podría no ser necesario. Es decir, se podrán cerrar todas las pestañas desde el extremo derecho.
> Más aún, si vale \\(\frac{a}{1} = b\\), entonces necesariamente \\(\frac{a}{m} > b\\) para \\(m > 1\\) (para cantidad de pestañas restantes mayor a 1), y también se podrán cerrar todas las pestañas desde la derecha, pues incluso cuando quede una pestaña su cruz estará en el extremo derecho.

Por la *Observación 1*, es claro que siempre deberemos mover el cursor al menos una vez. Además, se puede ver que en aquellos casos donde nuestra estrategia requeriría dos movimientos, no se pueden hacer menos, ya que eso implicaría hacer un solo movimiento. Es decir, elegir una posición donde se puedan repetir clicks hasta que no queden pestañas. Por lo tanto, la primera posición que elijamos debe ser la cruz de una de las pestañas iniciales, y podemos convencernos por las observaciones realizadas (y su comportamiento) que la elegida debe ser necesariamente la de la primera o la de la última. Así, por las observaciones se puede determinar que estaremos obligados a hacer dos cuando sea necesario.

> Si bien se podría realizar alguna demostración más formal, el punto de hacer olimpíadas no es estar demostrando todo el tiempo que lo que uno piensa está bien. Llegado a este punto del razonamiento, uno debería estar suficientemente convencido que lo descrito "funciona" y probarlo. El criterio para "convencerse" y cuándo hacerlo se desarrolla con la práctica. Si bien siempre existe la posibilidad de que uno se equivoque, con la práctica es menos frecuente que pase.

Habiéndonos convencido de esta estrategia, pasemos a la implementación. Sabemos que siempre se hará un movimiento, excepto el caso en que al principio valga \\(\frac{a}{m} > b\\), en el cual se harán dos movimientos. Sin embargo, tenemos que tener cuidado con el caso en que \\(\frac{a}{1} > b\\), para el cual todas las pestañas se cerrarán desde la derecha, y no hará falta un segundo movimiento del mouse. Por lo tanto, la respuesta siempre será un movimiento, excepto a que valgan simultáneamente \\(\frac{a}{m} > b\\) y \\(\frac{a}{1} \geq b\\). 

En programación siempre queremos evitar la comparación con números flotantes, por lo que podemos reescribir estas condiciones como `a > m*b` y `a > b`. Como las cotas del enunciado son \\(n \leq 10^9, b \leq 10^9\\), deberemos castear a `long long` para realizar el producto:

```cpp
#include<bits/stdc++.h>
 
using namespace std;
typedef long long ll;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int t; cin >> t;
    for (int tt = 0; tt < t; tt++) {
        int a,b,n; cin >> a >> b >> n;
        cout << 1 + (b*(ll)n > a && b < a) << '\n';
    }
   
    return 0;
}
```

> Observar que puede ser, según el enunciado, \\(n = 1\\). Incluso en este caso la solución funciona, porque la condición queda `b > a && b < a`, que nunca se puede cumplir. Así, siempre que \\(n=1\\), se devuelve `1`.

## D - Fibonaccharsis

Para poder resolver este problema, es necesario hacer una observación fundamental:

> **Observación 1**: Los valores de una secuencia de números *fibonacci-like* crecen rápidamente.

Consideremos la conocida *sucesión de Fibonacci*. Esta es, por definición, una secuencia de números *fibonacci-like* donde sus primeros dos números son \\(0\\) y \\(1\\). Podemos computar el \\(i\\)-ésimo término de la secuencia con un programa sencillo:

```cpp
#include<bits/stdc++.h>
 
using namespace std;

int const MAXN = 2e5+5;

int fibo[MAXN];

int main() {
    int n; cin >> n;

    fibo[0] = 0;
    fibo[1] = 1;
    for (int i = 2; i <= n; i++) {
        fibo[i] = fibo[i-1] + fibo[i-2];
        cout << i << ' ' << fibo[i] << '\n';
    }
   
    return 0;
}
```

Por lo tanto, como so una secuencia *fibonacci-like* está formada solo por números *no negativos* y es *no decreciente* según el enunciado, los valores más pequeños que se pueden elegir para \\(f_1\\) y \\(f_2\\) son los de la secuencia de Fibonacci. En consecuencia, si existe una secuencia de longitud \\(k\\) donde \\(n\\) sea el \\(k\\)-ésimo número de la secuencia (como pide el enunciado), entonces ésta debe ser de longitud **a lo sumo** \\(28\\).

> **Observación 2**: Existe **a lo sumo** una secuencia *fibonacci-like* con \\(f_k\\) y \\(f_{k-1}\\) fijos, pudiéndose determinar el resto de los valores de la secuencia incluyendo los de \\(f_1\\) y \\(f_2\\).

En efecto, si conocemos \\(f_k\\) y \\(f_{k-1}\\), luego \\(f_{k-2} = f_k - f_{k-1}\\), pues \\(f_k = f_{k-1} + f_{k-2}\\). Siguiendo el mismo razonamiento se podrían determinar \\(f_{k-3}, f_{k-4}, \dots, f_2, f_1\\). Sin embargo, esto no garantiza que los valores obtenidos sean los de una secuencia *fibonacci-like*:

* Por ejemplo, si \\(f_{k-1}\\) es menor a la **mitad** de \\(f_k\\), \\(f_{k-2}\\) deberá ser **mayor** a la mitad para que su suma sea \\(f_k\\), lo que viola la condición que una secuencia *fibonacci-like* es no decreciente.

* Además, como siempre restamos un número no negativo, es posible que los valores de \\(f_1\\) y \\(f_2\\) calculados sean negativos. 

A pesar de esto, como la igualdad **debe** mantenerse en secuencias *fibonacci-like*, es posible determinar si existe una secuencia de este tipo cuyos valores de \\(f_k\\) y \\(f_{k-1}\\) sean los fijados. Basta con calcular sucesivamente los valores \\(f_{k-3}, f_{k-4}, \dots, f_2, f_1\\) respetando la igualdad vista, y comprobar si la secuencia resultante solo contiene valores no negativos y es no decreciente.

De esta forma, "contar la cantidad de secuencias *fibonacci-like* que comienzan con \\(f_1\\) y \\(f_2\\) arbitrarios tal que \\(f_k = n\\)" es equivalente a "contar la cantidad de secuencias *fibonacci-like* con \\(f_k = n\\), \\(f_{k-1}\\) arbitrario".

Esto se explica ya que toda secuencia *fibonacci-like* válida que comienza con \\(f_1\\) y \\(f_2\\) arbitrarios determina los valores de \\(f_{k-1}\\) y \\(f_k\\), y viceversa. Por ende, sabiendo que \\(f_k = n\\), contar todos los \\(0 \leq f_{k-1} \leq f_k = n\\) es lo mismo que buscar lo que pide el enunciado. No se contarán secuencias de menos porque toda secuencia que se pide contar en el enunciado debe de tener valores \\(0 \leq f_{k-1} \leq f_k = n\\) por definición de secuencia *fibonacci-like*. Tampoco se contarán secuencias de más porque verificaremos en cada caso que las condiciones de una secuencia *fibonacci-like* se cumplan.

Combinando ambas observaciones, la complejidad de probar todos los valores de \\(f_{k-1}\\) posibles (\\(f_k\\) está fijo por enunciado) es \\(O(28 \cdot n)\\): probaremos todos los valores \\(f_{k-1} \in [0, n]\\), y calcular \\(f_1, f_2\\) para cada par \\(f_{k-1}, f_k\\) requiere recrear la secuencia, cuya longitud dijimos es a lo sumo \\(28\\).

Vale la pena recordar que el factor constante se suele excluir de la notación asintótica, por lo que simplemente podemos decir que la solución planteada tiene una complejidad \\(O(n)\\).

> En el código se utiliza una cota de \\(30\\) en vez de \\(28\\) por seguridad, ya que no es imposible que razonando la solución o implementándola cometamos un error "off-by-one", donde capaz la longitud máxima posible es \\(29\\) y no nos dimos cuenta. Dado que vamos a probar todas las secuencias de longitud \\(k\\) dado, el valor de \\(k\\) solo limita el tiempo que le lleva a nuestro programa calcularlas todas. Por una diferencia entre pequeña entre \\(28\\) y \\(30\\), es preferible que nuestro programa tarde "un poco más" en vez de accidentalmente no contar una secuencia válida por un error off-by-one. Al fin y al cabo, nuestro programa solo debe correr dentro del tiempo estipulado, no necesariamente ser el más rápido. Usualmente habrá un margen de tiempo suficiente en el tiempo límite establecido que nos permita hacer un cambio como este, especialmente porque se deben considerar diferencias entre los programas hechos por distintas personas (que pueden tener diferencias en sus tiempos de ejecución por otros motivos).

```cpp
#include<bits/stdc++.h>
 
using namespace std;

int const MAXK = 30;

int fiblike[MAXK];

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int t; cin >> t;
    for (int tt = 0; tt < t; tt++) {
        int n,k; cin >> n >> k;

        if (k >= MAXK) { // si es mayor a MAXK, ni hace falta fijarse porque es imposible
            cout << "0\n";
            continue; // siguiente caso
        }

        int cnt = 0;
        for (int i = 0; i <= n; i++) { // f_{k-1} es entre 0 y n
            fiblike[k-1] = n; // indexado en 0, por lo que la késima es en k-1
            fiblike[k-2] = i;

            bool posible = true;
            for (int j = k-3; j >= 0; j--) {
                fiblike[j] = fiblike[j+2] - fiblike[j+1];
                if (fiblike[j] < 0 || fiblike[j] > fiblike[j+1]) { // descarto cuando hay número negativo o deja de ser no decreciente
                    posible = false;
                }
            }

            if (posible) { // contamos las posibles
                cnt++;
            }
        }

        cout << cnt << '\n';
    }
   
    return 0;
}
```

## E - Mysterious Present (nivel 2/3)

> Para resolver este problema se requiere conocer sobre DP (*programación dinámica*). Si bien este problema no es de un nivel alto entre problemas de DP, se recomienda haber resuelto previamente algunos problemas de este tema antes de encarar este.

En primer lugar, es claro que si la tarjeta que quiere mandar Peter debe entrar en una cadena de sobres, no es de utilidad considerar aquellas cartas cuyas dimensiones sean en alguna dimensión menor o igual a la de la carta. Solo nos quedaremos con cartas cuyo largo y ancho sean ambos mayores a las de la tarjeta.

Para determinar la cadena más larga de cartas entre las consideradas, utilizaremos *programación dinámica*. Como es de costumbre al aplicar DP, es necesario tener un conjunto de estados que se deben recorrer en un orden determinado. ¿cómo son los estados? ¿en qué orden recorrerlos?

Definiremos nuestro estado como "la cadena más larga de cartas posible tal que la \\(i\\)-ésima en el orden es la última de la cadena". Es decir, el espacio de estados tendrá una sola dimensión: \\(dp[i]\\). Para las transiciones, será útil ordenar todas las cartas según _una de sus dimensiones_, desempatando por la otra dimensión en caso de igualdad. De esta forma, al calcular \\(dp[i]\\), sabemos que todas las cartas que podrían entrar dentro de la \\(i\\)-ésima ya fueron procesadas anteriormente. Así, podemos calcular la maxima longitud tal que la \\(i\\)-ésima sea la última de la cadena (\\(dp[i]\\)) como el *máximo* entre todas los estados ya procesados (\\(dp[j]\\), con \\(j < i\\)), pero solo aquellos correspondientes a cartas que _entren_ dentro de la \\(i\\)-ésima.

Simbólicamente, si notamos que una \\(carta_j\\) entra en otra \\(carta_i\\) como \\(carta_j \subset carta_i\\), la transición la DP está dada por:

$$dp[i] = \max_{\substack{j \, < \, i \\ carta_j \, \subset \, carta_i}} \; \{dp[j]+1\}$$

El enunciado no solo pide la longitud de la cadena, sino también mostrar una cadena válida. Para esto, se recomienda que cada vez se actualice un estado, mantener el "padre" de cada uno en una arreglo separado `P[MAXN]`. 

Finalmente, obtener la respuesta final consiste en recorrer todos los estados de la DP y quedarse con aquel que tenga la longitud más larga. Luego, se puede recorrer iterativamente por el padre de cada estado en la cadena hasta el primero de todos, almacenando las cartas utilizadas para posteriormente mostrarla en pantalla. Es necesario mantener los índices originales de las cartas en el ordenamiento, para hacer este recorrido posible.

```cpp
#include <bits/stdc++.h>
 
#define forn(i,n) for(int i = 0; i < int(n); i++)
#define forsn(i,s,n) for(int i = int(s); i < int(n); i++)
#define dforn(i,n) for (int i = int(n)-1; i >= 0; i--)
#define dforsn(i,s,n) for(int i = int(n)-1; i >= int(s); i--)
#define all(c) (c).begin(),(c).end()
#define pb push_back
#define fst first
#define snd second
#define FAST_IO ios::sync_with_stdio(false);cin.tie(nullptr);
 
using namespace std;
typedef vector<int> vi;
typedef long long ll;
typedef pair<int,int> ii;
 
const int MAXN = 5005;
 
struct env {
    int w,h,id;
 
    bool operator< (const env &o) const{
        if (o.w == w) return o.h > h;
 
        return o.w > w;
    }
};
 
env sobres[MAXN]; int ind = 0;
int dp[MAXN], P[MAXN];
 
int main() {
    FAST_IO;
 
    forn (i,MAXN) {dp[i] = 1; P[i] = -1;}
    int n,w,h; cin >> n >> w >> h;
 
    forn (i,n) {
        int W,H; cin >> W >> H;
        if (W <= w or H <= h) continue;
        sobres[ind++] = {W,H,i+1};
    }
 
    sort(sobres,sobres+ind);

    forn(i,ind) {
        forn(j,i) {
            if (sobres[j].w < sobres[i].w && sobres[j].h < sobres[i].h)
                if (dp[j]+1 > dp[i]) {
                    dp[i] = dp[j]+1;
                    P[i] = j;
                }
        }
    }
 
    int maxi = -1, myInd = -1;
    forn (i,ind) {
        if (dp[i] > maxi) {
            maxi = dp[i];
            myInd = i;
        }
    }
 
    if (maxi == -1) {cout << 0; return 0;}
 
    vi used;
    for (int i = myInd; i != -1; i = P[i])
        used.pb(sobres[i].id);
 
    cout << maxi << '\n';
    dforn(i,used.size()) cout << used[i] << ' ';
 
    return 0;
}
```

## F - Range Operation

A simple vista el problema parece algo complicado, pero tiene su truco.

Supongamos que se reemplazan los valores de un intervalo \\([l,r]\\) con \\((l+r)\\), ¿cuál es la nueva suma total del arreglo?

Representaremos con \\(S[l,r]\\) a la suma del arreglo original en el intervalo \\([l,r]\\):

$$nueva\_suma = S[0,l-1] + (l+r) \cdot (r-l+1) + S[r+1,n-1]$$

ya que la longitud del intervalo \\([l,r]\\) se calcula \\((r-l+1)\\). Aplicando distributiva en el producto y reorganizando vemos que:

$$nueva\_suma = S[0,l-1] + \cancel{l\cdot r} - l^2 + l + r^2 - \cancel{l \cdot r} + r + S[r+1,n-1]$$

$$nueva\_suma = (S[0,l-1] + l - l^2) + (r^2 + r + S[r+1,n-1])$$

Recordemos que nuestro objetivo es obtener la **máxima** suma. Por lo tanto, podemos iterar por una de nuestras "incógnitas" (\\(r\\)) y calcular para cada \\(r\\) la máxima suma del arreglo *si el intervalo reemplazado **terminara** en* \\(r\\). Luego la respuesta final será el máximo entre todas las máximas sumas del arreglo si el intervalo reemplazado terminara en \\(r\\), entre todos los valores de \\(r\\). En otras palabras, calculamos para cada \\(r\\), la mejor suma si se reemplaza \\([l,r]\\), para \\(l\\) **cualquiera**, y luego la respuesta es el máximo entre todos estos valores, siendo la mejor suma si se reemplaza \\([l,r]\\) para \\(l\\) **y** \\(r\\) cualesquiera.

En la expresión antes despejada, separamos la suma que queremos obtener en dos términos independientes: en uno de ellos solo está la incógnita \\(l\\), y en el otro solo está \\(r\\). Por ende, si fijamos \\(r\\) (porque vamos a probar para cada uno de sus valores), la máxima suma que se puede obtener reemplazando los valores de un intervalo \\([l,r]\\) por \\((l+r)\\) es la que se corresponde a un \\(l\\) tal que \\((S[0,l-1] + l - l^2)\\) sea máximo.

De esta forma, basta con recorrer el arreglo de izquierda a derecha sumando sus elementos (\\(S[0,i]\\)), e ir guardando el máximo valor de \\((S[0,i-1] + i - i^2)\\) en una variable. Luego, considerando el índice \\(i\\) como \\(r\\) (\\(r = i\\)), sabremos el primer término de la suma máxima del arreglo si el intervalo a reemplazar termina en ese \\(r\\). Faltaría tener a mano calculado \\((r^2 + r + S[r+1,n-1])\\) para poder completar el cálculo. En particular, \\(S[r+1,n-1]\\), ya que \\(r^2 + r\\) se puede calcular en la iteración. Si bien nosotros tenemos calculado el máximo \\((S[0,i-1] + i - i^2)\\), esto solo fue posible porque recorrimos todas las posiciones anteriores en iteraciones pasadas. Si venimos recorriendo de izquierda a derecha, no recorrimos todavía las posiciones hacia la derecha, por lo que no tenemos forma de calcular \\(S[r+1,n-1]\\) directamente.

Este problema se puede resolver de una forma sencilla. Si *anteriormente* recorremos el arreglo computando la suma **total** de sus elementos originales, podemos obtener el valor de \\(S[i+1,n-1]\\) (recordar \\(i = r\\)) restándole a dicha suma el valor de \\(S[0,i-1] + a_i = S[0,i]\\), ya que \\(suma\_total = S[0,i-1] + a_i + S[i+1,n-1] = S[0,i] + S[i+1,n-1]\\).

Con este último truco, se puede obtener la respuesta en dos pasadas al arreglo de entrada.

> Hay que tener cuidado que el enunciado dice que la operación se puede hacer **a lo sumo** una vez. Puede llegar a convenir no hacer la operación directamente, en cuyo caso la respuesta es la suma total de los elementos del arreglo original (que ya calculamos).

```cpp
#include<bits/stdc++.h>
 
using namespace std;
typedef long long ll;

int const MAXN = 2e5+5;

int arr[MAXN];

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int t; cin >> t;
    for (int tt = 0; tt < t; tt++) {
        int n; cin >> n;
        for (int i = 0; i < n; i++) {
            cin >> arr[i];
        }

        ll sumatotal = 0; // los valores pueden ser hasta 2 * 2e5, y hay hasta 2e5 -> 2e5 * 2e5 * 2 -> 8e10
        for (int i = 0; i < n; i++) {
            sumatotal += arr[i];
        }

        ll rta = sumatotal; // el enunciado dice que lo puedo hacer "at most once"
        ll sumacc = 0, maxleft = 0; // maxleft empieza en 0, por si conviene reemplazar un intervalo [0,r]
        for (int i = 0; i < n; i++) {
            // hasta acá sumacc -> S[0,i-1]

            // calculamos maxleft hasta este l inclusive
            maxleft = max(maxleft, sumacc + i+1 - (ll)(i+1)*(i+1)); // ojo usamos i+1 porque el ejercicio indexa en 1 no en 0, y OJO el producto (i es entero, castear a ll!)

            // respuesta para este r, máximo entre todos los l hasta r inclusive
            ll rtar = maxleft + (ll)(i+1)*(i+1) + i+1 + (sumatotal - (sumacc + arr[i])); // ídem ojo i+1 y ll
            
            // lo agregamos a la respuesta
            rta = max(rta, rtar);

            // actualizamos sumacc -> S[0,i]
            sumacc += arr[i];
        }

        cout << rta << '\n';
    }

    return 0;
}
```

## G - Same Difference

> **Observación**: Todos los caracteres deben ser cambiados al último de la string

Como la operación solo permite cambiar un carácter al siguiente de la string, es imposible cambiar el último carácter de la cadena. Dado que queremos hacer que todos sean iguales, estamos obligados a que todos sean iguales al último carácter que no puede cambiar.

Finalmente, como no tiene sentido cambiar un carácter a otro que no sea el último de la string, la cantidad de cambios será simplemente la cantidad de caracteres distintos del último de la string. En efecto, esto es posible simplemente recorriendo la cadena de derecha a izquierda, y aplicando la operación por cada carácter distinto. La operación siempre se podrá aplicar ya que el último de los caracteres de la string, llamémosle \\(c\\), es al que vamos a cambiar los que no sean iguales a \\(c\\):

* Se puede cambiar el último carácter de la string *que no sea \\(c\\)* porque por definición el siguiente a él debe de ser de ser \\(c\\).

* Luego, el último carácter de la string que no es \\(c\\) será otro en una posición menor, pudiéndole aplicar la operación bajo el mismo razonamiento que el ítem anterior. Si fuera éste, por ejemplo, el anterior al que aplicamos la operación en el ítem previo, se podrá aplicar la operación porque ahora ese vale \\(c\\).

* Se repetirá el proceso hasta que no queden caracteres que no sean \\(c\\).

> Si bien se puede recorrer la cadena en orden de derecha a izquierda, el enunciado sólo pide la cantidad de cambios. Como vimos, se producirá exactamente un cambio por carácter distinto de \\(c\\), por lo que es irrelevante el orden en que se contabilizan estos cambios.

```cpp
#include<bits/stdc++.h>
 
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int t; cin >> t;
    for (int tt = 0; tt < t; tt++) {
        int n; cin >> n;
        string s; cin >> s;

        int cnt = 0;
        char c = s.back(); // ultimo carácter de la string
        for (int i = 0; i < n; i++) {
            if (s[i] != c) {
                cnt++;
            }
        }

        cout << cnt << '\n';
    }
   
    return 0;
}
```
